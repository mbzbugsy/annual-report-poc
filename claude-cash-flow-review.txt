REVIEW VERDICT

Overall recommendation: REQUEST CHANGES

The extractor is well structured and follows the conventions already established by the income-statement and balance-sheet extractors. It correctly avoids fixed physical row dependency for KFA line items, never silently coerces missing or invalid values to zero, always preserves the two required semantic-review diagnostics, and produces a technically correct review_required extraction against the real workbook with no wiring into the PDF build pipeline. However, two substantive correctness/audit gaps were found (missing previous-period reconciliation, and no ambiguity guard on ÅR Layout composed-cell matching), along with two explicitly requested test scenarios that are not implemented. These should be addressed before this work is approved.


BLOCKERS

Finding number: 1
Severity: Blocker
Title: Reconciliation checks only cover the current period; previous-period subtotals are never validated
Affected file(s): src/cash_flow_extractor.py
Affected function/class: extract_cash_flow
Evidence: Inside extract_cash_flow, all five reconciliation inputs are built exclusively from the "valueCurrent" key, for example:
operating_expected = _sum_optional([_decimal_from_line(lines["operatingCashFlowBeforeWorkingCapital"], "valueCurrent"), _decimal_from_line(lines["changeInShortTermReceivables"], "valueCurrent"), _decimal_from_line(lines["changeInShortTermLiabilities"], "valueCurrent")])
operating_authoritative = _decimal_from_line(lines["operatingCashFlowTotal"], "valueCurrent")
The same pattern (valueCurrent only) is used for investing_expected/investing_authoritative, financing_detail_sum/financing_authoritative, net_expected/net_authoritative, and cash_bridge_expected/cash_bridge_authoritative. No equivalent computation exists anywhere in the file using "valuePrevious".
Why it matters: A broken or incorrect previous-period authoritative total (for example a shifted or mis-typed value in KFA column L) would produce no RECONCILIATION_DIFFERENCE diagnostic at all, since reconciliation never inspects the previous period. Previous-period values are part of the same authoritative contract as current-period values and should receive the same audit coverage. This does not prevent PDF rendering (the render guard already blocks all output), but it undermines the reliability of the reconciliation audit as a correctness signal for human reviewers.
Required change: Add previous-period equivalents of all five reconciliation entries (operatingSubtotal, investingSubtotal, financingSubtotal, netCashFlow, cashBridge), using the same _reconciliation_entry helper with "valuePrevious" values, and surface them in the reconciliation payload alongside the current-period entries.
Suggested test: test_reconciliation_previous_period_difference_produces_review_required (modify a previous-period authoritative cell, e.g. rows[22]["L"], and assert a RECONCILIATION_DIFFERENCE diagnostic referencing the previous period is produced)

Finding number: 2
Severity: Blocker
Title: ÅR Layout composed-cell matching can silently select the first ambiguous row
Affected file(s): src/cash_flow_extractor.py
Affected function/class: _find_layout_composed_cells
Evidence: Both the current-period and previous-period search loops break on the first formula match without checking for additional matches:
for row_index in sorted(ar_sheet.row_cells.keys()):
    cell = ar_sheet.row_cells[row_index].get("D")
    if cell is None or not cell.formula:
        continue
    if needle in normalized_formula:
        current_match = (row_index, cell)
        break
The equivalent KFA-side lookups (_find_rows_for_labels used by _extract_single_line and _component_entry) explicitly collect all matches and raise DUPLICATE_SEMANTIC_ANCHOR when len(matches) > 1. No such check exists for the ÅR Layout composed-cell lookup.
Why it matters: If ÅR Layout ever contained two rows whose formulas both reference the same KFA cell (for example a copy-pasted duplicate presentation row), the extractor would silently pick the first one with no diagnostic raised. This contradicts the general project principle that an ambiguous required mapping must never silently choose the first row. It does not currently manifest against the real workbook (only one match exists there), but the code has no structural protection against it.
Required change: Extend _find_layout_composed_cells to collect all matching rows per needle (current and previous) and raise a new diagnostic (for example AMBIGUOUS_AR_LAYOUT_COMPOSITION) with reviewRequired true when more than one match is found, mirroring the DUPLICATE_SEMANTIC_ANCHOR pattern used on the KFA side.
Suggested test: test_duplicate_ar_layout_anchor_produces_review_required (construct an ÅR Layout fixture with two D-column formulas both referencing KFA!J{acquisition_row}, assert the new ambiguity diagnostic is raised and the composed line is marked unresolved rather than picking the first row)

NON-BLOCKING OBSERVATIONS

Finding number: 5
Severity: Non-blocking
Title: Swedish character folding in label normalization is broader than necessary
Affected file(s): src/cash_flow_extractor.py
Affected function/class: _normalize_label
Evidence: _normalize_label lowercases, whitespace-folds, and then replaces both "å" and "ä" with "a", and "ö" with "o".
Why it matters: Folding both å and ä to the same character increases the theoretical risk of two distinct Swedish labels normalizing to the same string and being treated as equivalent anchors. This has not been observed to cause an actual collision in the current workbook's label set, but it is a wider fold than strictly required for whitespace/case-insensitive matching.
Required change: None required immediately. Consider narrowing normalization to case and whitespace folding only, or documenting explicitly why the diacritic fold is necessary.
Suggested test: test_normalize_label_does_not_collide_distinct_labels (optional, illustrative)

Finding number: 6
Severity: Non-blocking
Title: Period header extraction depends on fixed physical rows 7 and 8
Affected file(s): src/cash_flow_extractor.py
Affected function/class: _extract_period_metadata
Evidence: current_7 = kfa_sheet.row_cells.get(7, {}).get("J"), current_8 = kfa_sheet.row_cells.get(8, {}).get("J"), and the equivalent L-column previous-period lookups, all use hardcoded row numbers 7 and 8.

Required change: None required if confirmed intentional. If not intentional, consider adding a label-based or structural fallback.
